# Use bounded Interview Windows behind the revision barrier

Spec Grill V3 replaces V2's singleton Lookahead approval with a Brain-issued Interview Window containing zero to three pairwise-independent Question Permits; the deterministic Prepared Demo uses two. The Communicator still presents and clarifies exactly one permitted decision at a time, every Decision Summary still requires individual Product Manager confirmation, only one authoritative Brain request may run, and the validated complete revision applies before any queued work is revalidated or submitted. Each revision barrier expires the prior window; the Brain must return an exact disposition for every prior permit, either binding it to one fresh permit or invalidating it with a concise Brain-owned reason, while unused old permits disappear. Pairwise independence is validated only after the whole Question Roadmap passes referential-integrity and acyclicity checks. It requires unique roadmap-item, decision-key, and prompt identities; no direct or transitive dependency path between permitted items; binding to the exact in-flight operation; and no permit's bounded `ROADMAP-*` invalidation references naming itself or another permitted roadmap item. Domain keys may support diagnostics but are not proof. When all permits are exhausted during Brain work, the Communicator cannot invent or replenish questions; Persistent Brain Status and the last valid Specification remain visible until the Brain issues a fresh window. This trades some intrinsically stale Product Manager effort for lower idle time without introducing concurrent Specification merges, Communicator-authored decisions, or competing questionnaires. To bound wasted PM effort, consider only terminal PM-engaged Decision Summary jobs: two dependency-invalidated summaries among the last three eligible outcomes reduce the application window cap to one. Permitted deferrals, unused permits, pauses, cancellations, abandonment, provider failures, and batch-validation failures never enter that denominator. After shrinking, two consecutive eligible singleton Decision Summary jobs must apply without dependency invalidation before the application cap returns to three, while the Brain remains free to issue fewer or no permits. Across the frozen evaluation, three-permit windows fail the V3 quality gate if more than 25% of individually confirmed Decision Summaries become Not Applied because dependencies changed after at least 12 eligible terminal jobs, using the same exclusions.
